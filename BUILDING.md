# Building Barq JS

## What CI builds today

The [Build workflow](.github/workflows/build.yml) runs on every push / PR and is green:

| Job | What it proves |
| --- | --- |
| **barq-core engine (Linux)** | Compiles barq-core's `Storage` library from the vendored submodule with CMake — the native engine the SDK links against builds from source. |
| **Barq value types** | Strict `tsc` type-check + round-trip tests for the Barq value types (`ObjectId`, `UUID`, `Decimal128`, `Binary`, `EJSON`) that replace the `bson` dependency. |

The [Release workflow](.github/workflows/release.yml) publishes the pure-JS
`@barq/*` packages to npm on a `v*` tag.

## The native `@barq/barq` napi addon

The native node addon is **not yet buildable** and the reason is precise, not
hand-wavy: the JavaScript ⇄ C++ glue is generated by `barq-bindgen` from a
**binding spec** that describes barq-core's C++ object-store API. Two pieces are
currently missing from barq-core:

1. **The bindgen framework** (`barq-core/bindgen/`, the generator + `RelaxedSpec`
   model). barq-core's top-level `CMakeLists.txt` already looks for it
   (`if(EXISTS bindgen/CMakeLists.txt) add_subdirectory(bindgen)`), but the
   directory was removed during the fork.
2. **The spec** (`barq-core/bindgen/spec.yml`) describing barq-core's classes in
   the `barq::` namespace — including the reshaped token sync user
   (`barq::SyncUser` created from a token) and with the Atlas `App`/`Credentials`
   surface removed.

Everything on the JavaScript side is already in place for this:

- `packages/barq/bindgen/` holds the JS templates, `js_spec.yml`,
  `js_opt_in_spec.yml` and the `js-passes`.
- `packages/barq/package.json` wires `bindgen:wrapper` / `bindgen:jsi` to
  `bindgen/vendor/barq-core/bindgen/spec.yml`.
- The TypeScript layer (`packages/barq/src`) is already shaped for the token
  model and imports the Barq value types (no `bson`).

### To finish the native build

1. Restore `bindgen/` in **barq-core** (port it from the upstream engine, renamed
   to `barq::`) and add `bindgen/spec.yml` matching barq-core's current API.
2. Bump the barq-core submodule here to that commit.
3. Add a prebuild matrix job to `build.yml` that runs, per platform:
   `npm ci` → `npm run bindgen:wrapper -w @barq/barq` →
   `npm run prebuild-node -w @barq/barq` (and `prebuild-apple` / `prebuild-android`).

Until step 1 lands, the addon cannot be generated regardless of CI setup — it is
a barq-core task, not a barq-js configuration issue.

## Local build (engine only)

```sh
git submodule update --init --recursive
cmake -S packages/barq/bindgen/vendor/barq-core -B core-build \
  -DCMAKE_BUILD_TYPE=Release -DBARQ_BUILD_LIB_ONLY=ON -DBARQ_NO_TESTS=ON \
  -DBARQ_USE_SYSTEM_OPENSSL=ON
cmake --build core-build --target Storage --parallel
```
